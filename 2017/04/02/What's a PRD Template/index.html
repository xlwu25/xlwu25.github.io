什么是一个好的产品需求文档模版？

发表于 2017-04-02 | 分类于 需求分析

Loew 译

写一个好的产品需求文档（Product Requirements Document, PRD）不是一件简单的事情，但是很有必要。产品需求文档可以更好的帮助你交流新的功能与你的同事；还可以帮助你得出怎么样使用每个新的功能去解决问题和实现你的目标的答案。

这里列了一些产品需求文档必须具备的部分。每个部分都很重要。还有他们全来自于一个结合的整体，结果可以改变你的产品管理流程：

1.客观的（Objective）

2.核心成分（Core Components）

3.用户流（User Flow）

4.每个流步骤的细节（Details on Each User Flow Step）

5.分析（Analytics）

6.未来的功能（Future Features）

1.客观性

这是对你产品需求文档的一个介绍，你需要将一些背景信息告知给你的读者，从而可以帮助他们理解关键功能是什么。你可以让事情保持在高水平和回答问题，如：

  • 这个产品或功能给谁使用？
  • 你尝试去解决什么问题？
  • 这个功能的目的是什么？
  • 这个功能能做什么，不能做什么？

所以有客观的这个部分就像有一个「大哥哥」看着你，监督着你。它确保你建立的每个功能与产品团队必须实现的那个高水平目标相关联。它同样吸引你的注意力（还有文档读者的注意力）于核心产品或功能。

在本质上，它帮助你确保你的团队在正确的道路上和避免功能不明确（avoids feature creep）。

2.核心成分

考虑你的产品需求文档的核心成分作为目录。核心成分部分列出产品功能的关键部分。这将帮助文档读者理解接下来产品需求文档中的内容。

例如，你正在建立一个新的电子商务功能去帮助你的使用者购买一些彩色铅笔。这件事中的核心成分就可能是：

  • 搜索结果页面（你的用户怎么样去寻找他们想要的彩色铅笔）
  • 产品详细信息页面（关于他们的最喜爱的彩色铅笔的相关信息）
  • 合计账单页面（为了购买彩色铅笔所必须提供的信息）
  • 订单确认页面（他们购买彩色铅笔之后所发生的事）

在某种程度上，你的产品需求文档就像一个金字塔。你从金字塔顶部开始，定义你要解决的问题。当你在金字塔上向下移动（写产品需求文档），你会参与更多层细节在不同的每个功能成分中。核心成分部分会帮助文档读者更好理解哪个页面需要被建立，和什么东西被需要在建立每个页面时。

3.用户流

用户流是一个用来定义顾客、用户行程的空间。在核心成分和用户流之间或许会有一些重叠。但是只有用户流参与到用户如何弄通每个进程的步骤的详细实施中。当核心成分布置所有不同的页面和需求的功能，用户流是一步一步的用户们使用的为实现他们目标的进程。

再次使用彩色铅笔的例子，这件事中的用户流就可能是：

主页

  • 萨姆，一个彩色铅笔的爱好者，在搜索框中输入一个能带他去往搜索结果页面的短语。

搜索结果页面

  • 萨姆点击一支蓝色的彩色铅笔，为了得到更多的这支铅笔的信息。

产品细节页面

  • 萨姆决定购买这支彩色铅笔，然后点击了「添加到购物车」。
  • 然后萨姆点击了「去付款」。

收银台页面

  • 萨姆提交了购买表单在收银台页面。

订购确认页面

  • 萨姆查看订单确认页面

订单确认电子邮件

  • 萨姆收到一封订购确认电子邮件

在这个部分，我们从核心成分得出了页面事件然后在每个页面提取了一些关键动作。这里解释了用户流从开始到结束基本的流程。现在，你能画出功能和用户浏览的流程图了。你应该也理解了新功能的关键成分。

4.每个用户流步骤的细节

到目前为止，你已经定义了你的新功能的关键，客观性、核心成分和用户流。现在，是时候去定义每个页面事件的具体内容了。在每个页面事件中，你会概述工作的功能的所有元素——和肯呢个存在的所有不同状况。

第一，你应该列出所有不同元素在页面上。当你这么做了，记住你要编写的产品或功能的核心目标。任何你制作的元素应该直接服务于那个目的。

产品细节页面的元素
  • 价格
  • 产品名称
  • 描述
  • 社交媒体的分享
  • 评论回顾
  • 购买
  • 照片

对每个元素来说，你应该得知更多细节去帮助你的团队理解每个成分的运行，那个状况需要被解决，和哪个开发类型应该被提出。

这里是一个需要你提供信息的典型例子，还有当你写产品需求文档时，你需要问你自己的问题：

照片

  • 需要一些能够循环播放的照片在幻灯片播放器中。
  • 当萨姆点击照片，谈呢个看见更大的铅笔的照片。

价格

  • 萨姆（彩色铅笔热爱者）为买彩色铅笔会花的钱。

产品的名称

  • 这是特色在页面上。

描述

  • 描述应该写多少？
  • 有没有一个字符或者单词计数？

社会分享

  • 这个产品能被社交媒体分享吗？
  • 对于每个典型的社交媒体（脸书、Pinterest、推特、电子邮件等），反响会如何？
  • 好的提示：考虑下谁会看到这个产品和什么可以说服他们去点击。

评价回顾

  • 当没有人回顾评论怎么样做？
  • 当有一个或者更多的人回顾评论怎么样做？

当你完成这些在你写产品需求文档时，记得核心目的——为了让每个人保持在相同频道，所以他们能理解功能怎么样工作。为了达到这个目的，你要提供足够的细节，以至于任何看你文档的人都会完全理解这个产品的工作。

5.分析

在写你的产品需求文档时，你应该和你的团队一起工作，外加恰当的追踪去保证你的产品和功能的实现。没有恰当的追踪，你会两眼一抹黑，并且不能控制你的功能的发展。除非你对你的用户的工作知道得一清二楚，否则你不会知道怎么样去提高你的功能（或者功能中哪里可以提高）。

这个进程的第一步是列出3个主要的问题你想回答的。这些是问题，也是关键的度量标准，这将帮助你理解产品或功能是否做得好。

所以，对于电子商务功能，购买转化率就是一个关键的度量。这是你的访问者转化为彩色铅笔购买者的百分比。为了追踪这个信息，你可以查看访问每一步页面的人数在你的后台。

让我们看一下一个例子。这是一些虚假的访问每一步页面的人数的信息：

  • 搜索页面：100人
  • 产品细节页面：80人
  • 收银台页面：70人
  • 订单确认页面：10人

你注意到购买信息确认的人数和访问收银台页面的人数的关系了吗？相比较其他的页面，那是一个很大的跳跃。所以，这个数据指出你应该花更多的时间去使你的收银台页面更加优化。要努力去提高页面的实现，这会提高购买的百分比。

接下来是评估你的收银台去确认人们惦记了页面的哪些部分，还有是什么原因使他们没有完成购买。

在分析的时候，这些是你应该做的典型的问题和假设。一旦你的功能上线了，你应该关注这些问题和假说，认真去回答。

6.未来功能

这一部分会帮助你的团队理解产品会如何演化随着时间的过去。这个不需要非常深入的挖掘，只要帮助你的团队看清这个产品或功能通过你的视角。

记住作为一个项目经理，你需要以此成为专家。你应该比别人拥有更多产品知识，还有你应该分享这些知识去帮助你的团队达到相同的目的。

不要假想其他人知道你是做什么的——反之亦然。你能成为一个产品领导者的最大的助力是与你的团队分享知识。
